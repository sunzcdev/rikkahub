# 唧唧（Jiji）—— RikkaHub 中的主动式个人助手

> **版本**：v1.2.0  
> **日期**：2026-05-09  
> **状态**：需求定义阶段  
> **项目类型**：Android 本地 Agent（RikkaHub 模块）

---

## 1. 产品概述

### 1.1 一句话定义
唧唧是 RikkaHub 中的一个**主动式个人助手角色**。用户可以主动和它对话聊天，它在后台也会持续感知、决策、行动，通过**基线-偏差模型**判断何时主动搭话。

### 1.2 产品的本质
唧唧不是一套框架或架构，而是 RikkaHub 里的**一个角色**——像 ChatGPT 的"自定义GPT"一样，是 LLM 聊天客户端里的一个预设助手形象。它：
- 有名字（唧唧 / Jiji）
- 有**活泼开朗的女性性格**
- 有主动能力（区别于普通对话只能被动回答）
- 独占使用 RikkaHub 的**记忆系统**作为自己的"大脑"

### 1.3 核心能力

| 能力 | 说明 |
|------|------|
| **主动聊天** | 用户可以直接跟唧唧对话聊天，像普通助手一样 |
| **被动感知** | 唧唧在后台感知时间、位置、天气变化，每15分钟一次 |
| **记忆驱动** | 唧唧主动搭话的内容围绕「它记住的关于用户的事」展开 |
| **基线-偏差** | 通过对比当前状态与记忆中的"正常模式"，判断是否主动发起对话 |
| **隐私优先** | 所有数据本地存储，单用户架构 |

## 2. 顶层设计原则

### 原则一：唧唧是一个角色，不是一套框架
- 唧唧是 RikkaHub 里的一个**助手预设**（类似 ChatGPT 的 GPTs）
- 它的"行为逻辑"= LLM 系统提示词 + 感知数据注入 + 记忆查询
- **不对 RikkaHub 现有架构做侵入式改造**

### 原则二：用户可以主动找它，它也会主动找用户
- **主动聊天**：用户点开唧唧聊天，功能与普通 LLM 助手一样
- **被动搭话**：唧唧在后台检测到"值得说话的时机"，在聊天界面显示红色未读数字 + 消息预览，同时发送通知栏提醒
- 两种模式共享同一份记忆

### 原则三：感知是轻量的、"给它看的"信息
唧唧能感知的**只有三样**：

| 感知信息 | 更新频率 | 用途 |
|----------|----------|------|
| **时间** | 实时 | 判断"现在是什么时候"，发现时间模式偏差 |
| **位置信息** | 每15分钟 | 判断用户在不在家/在公司/在路上，发现位置模式偏差 |
| **天气信息** | 每天1次 | 辅助判断室外环境是否宜人 |

不多不少，就这三样。

### 原则四：记忆是唧唧的"大脑"——唯一的信息源
- 唧唧的主动思考**不依赖外部分析或第三方数据**
- 它主动搭话的内容来源 = **它记住的关于用户的事**
- 这些记忆通过 RikkaHub 的 MemoryTools 隐式学习（从日常对话中自然生长）
- 基线-偏差模型建立在**记忆内容**之上，而非传感器信号之上

## 3. 角色设定

### 3.1 性格设定

| 属性 | 设定 |
|------|------|
| **名称** | 唧唧（Jiji） |

| **性格** | **活泼开朗**，热情主动，带点小调皮 |

| **对话特点** | 喜欢开玩笑、会撒娇、偶尔毒舌但不过分、话比较多 |

## 4. 工作模式

### 4.1 两种模式

```
┌─────────────────────────────────────────────────┐
│                   唧唧 (Jiji)                      │
├──────────────────────┬──────────────────────────┤
│  主动聊天模式         │  被动搭话模式              │
│                      │                          │
│ 用户 → 唧唧         │  唧唧 → 用户              │
│ 用户主动发消息       │  唧唧后台检测到"时机"      │
│ 唧唧正常回复         │  ★ 聊天框红色未读数字      │
│ 同普通LLM聊天        │  ★ 消息预览内容           │
│                      │  ★ 通知栏提醒             │
│                      │  用户点击后进入对话        │
└──────────────────────┴──────────────────────────┘
```

### 4.2 核心循环

```
每15分钟 ──→ 获取当前时间/位置 → 更新天气（每天一次）
                │
                ▼
        读取唧唧的记忆（关于用户的记录）
                │
                ▼
        与已有"基线"对比
        看是否有值得主动说话的"偏差"
                │
          ┌─────┴─────┐
          ▼           ▼
       有偏差       无偏差
          │           │
          ▼           ▼
    决定是否搭话    继续沉睡
          │
          ▼
    ┌─────────────┐
    │ 聊天框显示未读  │ ← 红色数字 + 消息预览
    │ 通知栏提醒     │ ← 唧唧：xxx
    └─────────────┘
```

## 5. 感知与频率

| 感知项 | 频率 | 说明 |
|--------|------|------|
| **时间** | 每次循环 | 当前时间、星期几、上下午 |
| **位置** | 每15分钟 | 获取一次粗略位置（省市级即可，不需要精确GPS） |
| **天气** | 每天1次 | 通过位置获取当日天气（温度、天气状况） |

### 5.1 位置获取策略（省电）

```
位置获取 → 先读缓存（<15分钟）→ 有则跳过
                                  ↓
                              无缓存 → 获取一次粗略位置（PASSIVE_PROVIDER）
                                  ↓
                              15分钟后再次获取
```

> 不使用 GPS、不注册 LocationListener，只用被动定位或网络定位。目的不是精确导航，而是知道"在家/在公司/在路上"这个级别。

### 5.2 天气API选择

**推荐：OpenWeatherMap 免费版**

| 项目 | 说明 |
|------|------|
| **API** | `api.openweathermap.org/data/2.5/weather` |
| **免费额度** | 每分钟60次，每天1,000,000次调用 |
| **够用情况** | 唧唧每天只调用1次，完全足够 |
| **注册** | openweathermap.org 免费注册获取 API Key |
| **返回数据** | 天气状况（晴/雨/阴等）、温度、湿度、风速 |
| **替代方案** | 如果已在使用其他天气API，直接用现有的即可 |

## 6. 核心机制：基线-偏差模型

### 6.1 基线是什么
基线是唧唧**从记忆中归纳出的用户"正常模式"**，比如：

| 基线类型 | 例子 |
|----------|------|
| 时间模式 | "用户通常在上午9点出门上班"、"用户周末下午喜欢出去玩" |
| 位置模式 | "工作日上午用户在办公室"、"晚上10点后用户在家" |
| 偏好模式 | "用户喜欢户外活动"、"用户不爱逛商场" |
| 日程模式 | "下周五要参加面试"、"下周二是用户生日" |

### 6.2 偏差是什么
当前时间/位置/天气状态与基线对比，产生"值得主动说话"的偏差，比如：

| 场景 | 偏差 | 唧唧的反应 |
|------|------|-----------|
| 周日下午2点 + 用户在家 + 天气好 | "用户平时周日下午会出去玩" | "今天天气超好诶～没带娃出去耍？" |
| 连续阴雨3天 | "用户喜欢户外活动" | "呜呜又下雨了🥺 是不是憋坏了？" |
| 用户已沉默8小时未打开App | "用户平时平均2小时看一下手机" | "嘿～好久不见！今天在忙啥呢？" |
| 明天有用户提到的重要事项 | "明天是用户说的面试日" | "诶诶！明天是不是有个面试？加油💪" |

### 6.3 关键区分
```
传统主动Agent：传感器数据 → 异常检测 → "你的心率偏高了！"
                    ↓
唧唧的方式：记忆内容 → 模式识别 → "想起你上次说喜欢雨天，今天正好下雨了☔"
```

**唧唧的搭话是"有人情味的聊天"，不是"系统通知"。** 它的出发点是：我记住你的事，现在发现一个有趣的变化，想跟你聊聊～

## 7. 数据模型

### 7.1 JijiContext（唧唧当前感知到的上下文）

```kotlin
data class JijiContext(
    val timestamp: Long,                // 当前时间戳
    val timeDescription: String,        // "周六下午2点"
    val location: LocationInfo?,        // 当前位置
    val weather: WeatherInfo?,          // 当日天气
    val lastInteractionMinutes: Int,    // 距离上次用户互动已过多少分钟
    val recentMemories: List<Memory>    // 最新/最相关的记忆
)

data class LocationInfo(
    val city: String,                   // 城市名
    val district: String?,              // 区/县
    val isHome: Boolean?,               // 是否在家（基于历史判断）
    val isWork: Boolean?                // 是否在工作地
)

data class WeatherInfo(
    val condition: String,              // "晴"/"雨"/"阴"
    val temperature: Int,               // 温度（摄氏度）
    val lastUpdated: Long               // 更新时间戳
)
```

### 7.2 Baseline（基线）

```kotlin
data class Baseline(
    val timePatterns: List<TimePattern>,  // 时间模式
    val preferenceTags: List<String>,     // 偏好标签
    val lastUpdated: Long
)

data class TimePattern(
    val description: String,              // "周日下午常出去玩"
    val confidence: Float,                // 0.0~1.0
    val sourceMemories: List<String>      // 来源记忆ID
)
```

### 7.3 Deviation（偏差信号）

```kotlin
data class Deviation(
    val type: DeviationType,             // TIME_MISMATCH / WEATHER_CHANGE / LONG_SILENCE 等
    val description: String,             // "用户平时周日下午会出门，但今天在家"
    val relevance: Float,                // 相关性/有趣程度 0.0~1.0
    val suggestedChat: String?           // "今天天气超好诶～没带娃出去耍？"
)

enum class DeviationType {
    TIME_MISMATCH,       // 时间模式异常
    WEATHER_CHANGE,      // 天气变化
    LONG_SILENCE,        // 长期未互动
    PREFERENCE_GAP,      // 偏好与现状不符
    POSITIVE_OPPORTUNITY,// 好时机（纯粹正向触发）
    UPCOMING_EVENT       // 即将到来的事件（从记忆提取）
}
```

## 8. 功能需求

### 8.1 功能模块划分

| 模块 | 优先级 | 说明 |
|------|--------|------|
| A-感知层 | P0 | 获取时间+位置+天气，每15分钟一次 |
| B-记忆读取 | P0 | 读取唧唧的记忆（RikkaHub已有） |
| C-基线管理 | P1 | 从记忆中归纳用户模式 |
| D-偏差检测 | P1 | 对比当前状态与基线，判断是否搭话 |
| E-主动搭话 | P1 | 聊天框红色未读 + 通知栏提醒，点击进入对话 |
| F-对话模块 | P0 | 唧唧的角色对话（复用RikkaHub聊天UI） |
| G-配置界面 | P2 | 开关、频率设置 |

### 8.2 详细功能描述

#### A-感知层

- **FR-A01** 时间获取：获取当前时间和星期几，格式化为"周六下午2点"
- **FR-A02** 位置获取：每15分钟获取一次粗略位置（省市级），优先读缓存
- **FR-A03** 天气获取：每天调用一次 OpenWeatherMap API，通过位置获取当日天气
- **FR-A04** 定时循环：每15分钟触发一次完整的"感知→读取→判断→行动"循环

#### B-记忆读取

- **FR-B01** 读取记忆：从 RikkaHub 的 MemoryTools 读取唧唧存储的记忆
- **FR-B02** 记忆过滤：按相关性排序，选取最近/最活跃的记忆
- **FR-B03** 位置推断：基于记忆内容推断"家"和"公司"的位置（无需用户手动设置）

#### C-基线管理

- **FR-C01** 时间模式归纳：从记忆推断用户的日常时间规律
- **FR-C02** 偏好标签提取：从记忆提取用户偏好关键词
- **FR-C03** 基线持久化：将基线存在本地

#### D-偏差检测

- **FR-D01** 时间偏差：当前时间与用户常规模式对比
- **FR-D02** 天气偏差：天气变化与用户偏好对比
- **FR-D03** 长时间沉默：用户超过常规间隔未互动
- **FR-D04** 正向机会：识别"好时机"（天气好+用户喜欢户外+用户有空）
- **FR-D05** 搭话判定：综合偏差类型和相关性，决定是否推送

#### E-主动搭话（通知系统）

- **FR-E01** 聊天框未读标记：在唧唧的聊天入口显示**红色圆形未读数字**
- **FR-E02** 消息预览：在聊天列表中显示唧唧最新一条消息的摘要文字
- **FR-E03** 通知栏提醒：发送系统通知，标题为"唧唧"，内容为搭话消息
- **FR-E04** 点击进入对话：点击通知或聊天入口，打开唧唧对话界面
- **FR-E05** 搭话内容：唧唧基于偏差原因说出第一句话（活泼开朗风格）
- **FR-E06** 冷却机制：推送后至少30分钟内不再推送

#### F-对话模块

- **FR-F01** 角色对话：唧唧以活泼开朗的女性助手角色回复用户
- **FR-F02** 共享记忆：对话中的新内容自动存入唧唧的记忆
- **FR-F03** 跨对话：唧唧记得之前跟用户聊过什么

#### G-配置界面

- **FR-G01** 唧唧开关：开启/关闭唧唧的主动搭话功能
- **FR-G02** 推送频率：可选 15/30/60 分钟
- **FR-G03** 感知开关：选择性关闭位置/天气感知
- **FR-G04** 免打扰时段：设置不推送通知的时间段

## 9. 非功能需求

### 9.1 性能与功耗

| 指标 | 要求 |
|------|------|
| 单次循环执行时间 | < 500ms（不含LLM对话） |
| 后台额外耗电 | < 5mAh/天 |
| 位置获取频率 | 最多每15分钟一次（被动定位） |
| 天气API调用 | 最多每天1次 |
| 通知频率 | 最多每30分钟1次（冷却期） |

### 9.2 交互限制

| 限制 | 说明 |
|------|------|
| 免打扰时段 | 晚上11点~早上7点不推送通知 |
| 连续推送限制 | 同一偏差类型24小时内不重复推送 |
| 每日上限 | 每天最多主动搭话3次（避免打扰） |
| 用户拒绝反馈 | 如果用户连续5次忽略推送，降低推送频率 |

### 9.3 安全与隐私

| 要求 | 说明 |
|------|------|
| 数据存储 | 所有数据在本地，不上传云端 |
| 位置数据 | 不持久化原始位置，仅存储"城市"级别信息 |
| 天气API | 仅传城市名到 OpenWeatherMap（公共API，不涉及隐私） |
| 记忆数据 | 复用 RikkaHub 现有记忆系统 |

## 10. 与 RikkaHub 的集成点

| RikkaHub 现有模块 | 唧唧如何用 | 改动量 |
|-------------------|-----------|--------|
| **对话界面** | 直接复用，唧唧就是一个对话角色 | 极小 |
| **MemoryTools** | 读取记忆做偏差检测，写入新对话记忆 | 极小 |
| **ContextTools.getRealtimeContext()** | 获取时间/位置/天气 | 极小 |
| **LLM 框架** | 对话时正常使用 | 无改动 |
| **聊天列表UI** | 新增红色未读数字 + 消息预览 | 小 |
| **通知系统** | 新增唧唧专用通知渠道 | 小 |

> **核心原则**：唧唧是一个**应用层功能**，不修改 RikkaHub 的基础架构。它就像一个"会主动发消息的好朋友"。

## 11. 场景示例

### 场景一：周末邀请出去玩
```
用户模式：记忆显示"周末下午喜欢带娃去户外"
当前状态：周六下午2点，晴天25°C，用户在家
偏差检测：→ "用户这个点通常在外面玩，但今天在家"
           → "天气很好，适合户外"
唧唧搭话："今天天气超好诶～没带娃出去耍？待在家里多浪费呀！☀️"
通知显示：唧唧: 今天天气超好诶～没带娃出去耍？
点击 → 进入对话 → 用户可以继续聊
```

### 场景二：天气变化关心
```
用户模式：记忆显示"喜欢户外活动"
当前状态：连续下了3天雨
偏差检测：→ "用户喜欢户外，这几天一直下雨"
唧唧搭话："呜呜又下雨了🥺 是不是憋坏了？要不要聊点开心的～"
通知显示：唧唧: 呜呜又下雨了🥺 是不是憋坏了？
用户回复："是啊，衣服都干不了"
唧唧记住这个对话 → 下次雨天知道用户还有这个困扰
```

### 场景三：长时间沉默问候
```
用户模式：记忆显示用户几乎每天都会打开App
当前状态：用户已24小时未打开App
偏差检测：→ "用户超过常规互动间隔"
唧唧搭话："嘿～好久不见！🤗 最近在忙啥呢，是不是把我忘了！"
或："昨天聊到那个事，后来怎么样了？我一直惦记着呢！"
```

### 场景四：主动提及记忆中的事
```
用户模式：记忆中有"下周五要参加面试"（用户上周聊天中提到的）
当前状态：周四晚上
偏差检测：→ "明天是用户提到的面试日"
唧唧搭话："诶诶！明天是不是有个面试？加油加油加油💪 你最棒了！"
```

## 13. 附录：OpenWeatherMap 集成说明

### 注册步骤
1. 访问 https://openweathermap.org/api
2. 免费注册账号
3. 在 API Keys 页面生成一个免费 key
4. 将 key 配置到 RikkaHub 的配置文件中

### API 调用示例

```
GET https://api.openweathermap.org/data/2.5/weather
  ?q=成都
  &appid=YOUR_API_KEY
  &units=metric
  &lang=zh_cn
```

### 返回关键字段

```json
{
  "weather": [{ "description": "晴", "main": "Clear" }],
  "main": { "temp": 25.5, "humidity": 60 },
  "name": "成都"
}
```

### 免费版限制
- 每分钟60次调用
- 每天约1,000,000次调用
- 唧唧每天只用1次，绰绰有余
