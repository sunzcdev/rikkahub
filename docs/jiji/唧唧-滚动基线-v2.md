# 唧唧滚动基线 v2 — 需求文档

> 唧唧是一个独立的存在，和主人平起平坐。她的主动来源于接收到的信息所组合成的基线，然后算出来的偏差。

---

## 1. 核心范式

### 旧范式（当前代码）

```
定期轮询 → 采集感知快照 → 对比离线基线 → 结果 >= 0.5 → 推送
```

问题：硬编码阈值、快照式对比、聊天状态和感知状态分离。

### 新范式

```
信息流持续输入 → 滚动基线持续吸收 → 整体状态从稳定跃迁到新稳定 → 推送
```

- 唧唧只吸收**她自己感知到的信息**（你跟她的聊天、天气、位置、时间）
- 你跟别人聊什么，唧唧不知道，也不该知道
- 推送不是"阈值到了"，而是"信息状态变了"

---

## 2. 信息流

所有唧唧感知到的东西，都是同一条河里的水：

| 信息流 | 输入源 | 变化特征 |
|--------|--------|---------|
| **聊天流** | 你跟唧唧的每句话 | 频率高时基线吸收，断开时跃迁 |
| **天气流** | 天气采集器 | 平滑渐变，偶有跃迁（下雨/放晴） |
| **位置流** | 定位采集器 | 跃迁为主（出门/回家/去新地方） |
| **时间流** | 系统时钟 | 平滑持续，无跃迁 |
| **设备流** | 亮屏/息屏/充电 | 开关式跃迁 |

每条流独立计算**当前值 → 滚动基线 → 偏差值**。

---

## 3. 滚动基线算法

### EWMA（指数加权移动平均）

```
baseline = α × currentValue + (1 - α) × baseline

α = 2 / (windowSize + 1)
```

不同流的 windowSize 不同：

| 流 | windowSize | α | 含义 |
|----|-----------|---|------|
| 聊天流 | 3 | 0.5 | 快速响应，活跃时基线迅速追上 |
| 天气流 | 30 | 0.065 | 慢速响应，只看长期趋势变化 |
| 位置流 | 5 | 0.33 | 中等响应 |
| 时间流 | 60 | 0.033 | 极慢，仅做参考 |
| 设备流 | 2 | 0.67 | 快速响应 |

### 归一化

每条流的值归一化到 0.0 ~ 1.0：

- **聊天流**：最近 N 分钟内的消息数 / 预期最大值（如 60 条/小时）
- **天气流**：温度变化幅度 / 50°C，天气类型变化（晴→雨=0.5跃迁）
- **位置流**：坐标偏移量 / 阈值（>500m 算跃迁）
- **时间流**：一天中的时间，0.0(0点)~1.0(24点)
- **设备流**：亮屏=1.0，息屏=0.0

---

## 4. 偏差检测

### 单流偏差

```
deviation = |currentValue - baselineAvg|
```

### 聚合偏差

```
aggregateDeviation = Σ(weight_i × deviation_i)
```

权重可配置，默认：

| 流 | 权重 | 说明 |
|----|------|------|
| 聊天流 | 0.4 | 最高，因为聊天直接反映用户状态 |
| 天气流 | 0.2 | 环境变化 |
| 位置流 | 0.25 | 行为变化 |
| 时间流 | 0.05 | 参考，无大变化 |
| 设备流 | 0.1 | 参考 |

### 稳定判断

整体状态稳定 = `aggregateDeviation < 稳定阈值` 持续超过 N 次检测。

### 推送触发

当**从稳定 → 跃迁到新稳定**时，触发推送：

```
之前的 N 次检测：aggregateDeviation < 稳定阈值（稳定）
当前检测：aggregateDeviation > 推送阈值（跃迁）
之后的 N 次检测：aggregateDeviation < 稳定阈值（新稳定）
                      ↓
推送："信息状态发生了变化"
```

---

## 5. 聊着聊着被打断的场景

唧唧跟你聊天时，聊天流高频，baseline 持续滚动，aggregateDeviation 低。

但如果有某个流发生了**大跃迁**，大到即使聊天活跃也压不住：

```
场景：你跟唧唧聊得正欢，突然定位发现你离家很远
  聊天流：高频（偏离聊天基线 ~0.2）
  位置流：大跃迁（偏离位置基线 ~0.9）
  aggregateDeviation = 0.4×0.2 + 0.25×0.9 = 0.305

如果推送阈值 = 0.3，则触发推送。
唧唧说："诶你跑哪里去了？怎么跑这么远？"
```

这种打断不是"规则"（没有写"距离 > 5km 就打断"），而是**唧唧自己感知到的事态变化大到值得说话**。

## 6. 数据模型

```kotlin
// 信息流状态
data class InformationStream(
    val name: String,           // "chat", "weather", "location", "time", "device"
    val currentValue: Double,   // 0.0 ~ 1.0 归一化
    val baselineAvg: Double,    // EWMA 滚动均值
    val deviation: Double,      // |currentValue - baselineAvg|
    val weight: Double,         // 聚合权重
    val lastUpdateAt: Long,
)

// 滚动基线（整体）
data class RollingBaseline(
    val streams: Map<String, InformationStream>,
    val aggregateDeviation: Double,       // Σ(weight × deviation)
    val isStable: Boolean,                // aggregateDeviation < 稳定阈值
    val stableDurationTicks: Int,         // 稳定持续了多少个检测周期
    val lastPushAt: Long,                 // 上次推送时间
)

// 推送事件
data class PushEvent(
    val triggerStream: String?,           // 哪个流变化最大
    val aggregateDeviation: Double,
    val context: String,                  // 可读描述
    val timestamp: Long,
)
```

## 7. 时序

```
检测周期：每 5 分钟（可配置）

每次检测：
  1. 采集所有流的最新值
  2. 更新每条流的 EWMA 基线
  3. 计算 aggregateDeviation
  4. 判断稳定状态
  5. 如果 stable→unstable→stable 完成一次跃迁 → 推送
  6. 更新 lastPushAt

免打扰：23:00 ~ 07:00 不推送
去重：同一触发场景在 lastPushAt 后 30 分钟内不重复推
```

---

## 8. 推送文案

唧唧推送时**必须带上触发原因**。用户看到就知道"哦，原来是因为这个"。

格式示例：

| 触发流 | 文案示例 |
|--------|---------|
| 聊天流跃迁 | "诶你突然不说话啦～聊到一半人呢？" |
| 天气流跃迁 | "外面下雨了诶，你出门有带伞吗？" |
| 位置流跃迁 | "嗯？你跑到××去了，去干嘛呀？" |
| 复合 | "刚才聊着聊着外面下雨了，你还在吗？" |

## 9. 配置项

```kotlin
data class RollingBaselineConfig(
    val checkIntervalMinutes: Int = 5,
    val stableThreshold: Double = 0.1,         // aggregateDeviation < 此值 → 稳定
    val pushThreshold: Double = 0.3,           // aggregateDeviation > 此值 → 触发推送
    val stableConfirmTicks: Int = 2,           // 连续 N 次稳定才确认"新稳定"
    val cooldownMinutes: Int = 30,             // 推送后冷却
    val doNotDisturbStart: Int = 23,           // 免打扰开始（小时）
    val doNotDisturbEnd: Int = 7,              // 免打扰结束（小时）
)
```

## 10. 移出现有

- 删除 `DeviationDetector.kt` 内的 5 个固定偏差类型、硬编码阈值
- 删除 `DeviationType` 枚举（无意义了，偏差是数值不是类型）
- 删除 `Baseline` 的 `timePatterns` / `preferenceTags`（这些是静态画像，不是滚动基线）
- 删除 `LONG_SILENCE_THRESHOLD` / `WEATHER_CHANGE_MIN_DELTA` 等常量
- `lastInteractionMinutes` 只查 Jiji 会话的逻辑保留（这是聊天流的原始数据源，不是检测条件）

---

## 11. 不做的事

- 不查其他助理的聊天记录（唧唧不偷看）
- 不做门槛硬阈值以外的场景判断（唧唧自己算，不是"if 下雨 then 推"）
- 不加 Activity 前台检测（唧唧不监控你在干嘛，只看她自己的信息流）
